# 待办任务页面设计

## 概述

本设计为当前 Next App Router 项目新增一个项目任务管理页面。项目保留现有流程图首页，在此基础上新增 `/todos` 路由，并在全局顶部导航中提供“流程图页”和“待办页”两个入口，支持双向切换。

第一版目标是单用户、项目内自带真实后端存储、适合本地开发的项目任务页。页面采用“控制台列表”布局，强调任务管理效率，而不是看板式展示。

## 目标

- 保留现有 `/` 流程图页，不改动其核心功能
- 新增 `/todos` 页面，作为独立任务管理入口
- 在顶部导航中提供跨页面切换能力
- 使用项目内轻量数据库完成真实数据持久化
- 支持创建任务、切换状态、设置优先级、按固定分类筛选
- 页面文案和交互统一使用中文

## 非目标

- 不实现用户系统、登录和权限控制
- 不实现多人协作、任务共享和任务分配
- 不实现拖拽排序、复杂编辑弹层和看板布局
- 不实现截止日期、负责人、项目分区和标签系统
- 不重构当前流程图首页的业务结构

## 信息架构

### 路由

- `/`
  - 保留现有 Skill Flow Parser 页面
- `/todos`
  - 新增待办任务页面

### 全局导航

导航放入共享布局中，始终显示在页面顶部，包含：

- `流程图页`，跳转到 `/`
- `待办页`，跳转到 `/todos`

当前页面对应导航项需要有明显高亮态，方便用户理解当前所在位置。

## 页面结构

`/todos` 页面采用自上而下的控制台列表布局。

### 标题区

- 页面标题，例如“待办任务”
- 一句简短说明，例如“管理当前项目中的日常任务和进度”

### 摘要区

使用卡片展示关键统计，第一版至少包含：

- 全部任务数
- 进行中任务数
- 已完成任务数

摘要信息来自数据库实时查询结果，而不是前端硬编码。

### 工具区

提供以下操作入口：

- 分类筛选
- 优先级筛选
- 状态筛选
- `新建任务` 操作

第一版的“新建任务”采用内联展开表单，而不是弹窗。这样可以减少交互层级，适配高频录入场景，并降低首版实现复杂度。

### 列表区

以高密度列表形式展示任务，每行直接展示：

- 标题
- 状态
- 优先级
- 分类

每行支持快速状态切换，不需要进入详情页才能完成基础管理操作。

## 组件边界

建议将页面拆分为以下组件：

- `TopNav`
  - 顶部导航组件，放入共享布局
- `TodoSummaryCards`
  - 展示摘要统计
- `TodoFilters`
  - 展示和控制筛选条件
- `TodoCreateForm`
  - 内联新建任务表单
- `TodoList`
  - 任务列表容器
- `TodoListItem`
  - 单行任务项

拆分原则：

- 导航与业务页面解耦，便于复用
- 筛选、创建、列表各自负责单一职责
- 任务项保持轻量，避免首版即引入复杂编辑逻辑

## 数据设计

### 技术选型

- 数据库：`SQLite`
- ORM：`Prisma`
- 路由模式：`Next App Router`
- 数据读写方式：服务端读取 + Server Action 写入

选用原因：

- 适合单用户、本地开发、自带数据文件的场景
- Prisma 模型定义和迁移流程清晰
- 与 App Router 的服务端读取模式契合

### 数据模型

第一版仅定义一个 `Todo` 模型，字段如下：

- `id`
- `title`
- `status`
  - 枚举值：`todo` / `in_progress` / `done`
- `priority`
  - 枚举值：`high` / `medium` / `low`
- `category`
  - 固定值集合：`开发` / `产品` / `设计` / `运营`
- `createdAt`
- `updatedAt`

### 数据约束

- `title` 必填，不能为空白字符串
- `status`、`priority`、`category` 必须落在允许范围内
- 第一版不引入软删除字段
- 第一版不引入额外关系表
- 第一版不提供默认种子数据，初次进入页面时展示空状态

## 数据流与后端交互

### 页面读取

- 访问 `/todos` 时，由服务端直接从数据库读取任务列表和摘要数据
- 页面初始渲染即使用真实数据库数据

### 筛选

筛选条件通过 URL query 表达，至少支持：

- `status`
- `priority`
- `category`

这样做的目的是：

- 刷新页面后保留筛选状态
- 允许通过 URL 直接打开特定筛选结果
- 保持筛选逻辑与服务端读取逻辑一致

### 写操作

以下操作使用 Server Action：

- 创建任务
- 更新任务状态

写入成功后，重新获取当前页面的数据，使：

- 任务列表展示最新数据
- 摘要数字同步更新
- 页面不依赖前端本地假状态维持一致性

## 页面交互设计

### 新建任务

点击 `新建任务` 后展开内联表单，字段包括：

- 标题
- 优先级
- 分类

默认行为：

- 新建任务默认状态为 `todo`
- 提交成功后清空表单并收起

### 状态切换

每个任务项支持直接切换状态，例如：

- `待开始`
- `进行中`
- `已完成`

第一版优先保证状态切换路径短、反馈清晰，不增加复杂编辑入口。

### 空状态

需要区分两类空状态：

- 数据库当前没有任何任务
- 当前筛选条件下没有匹配结果

两种情况的文案需要不同，避免用户误解为页面加载失败。

## 错误处理

### 表单错误

- 标题为空时阻止提交
- 在表单区域内显示明确中文错误提示

### 写入失败

- 数据库写入失败时显示通用错误提示
- 不允许用户点击后无反馈

### 读取失败

- 列表读取失败时，在列表区域显示错误状态
- 提供明确重试路径

### 非法输入

- 服务端再次校验 `status`、`priority`、`category`
- 即使前端控件正常，也不信任客户端提交值

## 视觉与内容方向

页面风格延续当前项目的暖中性色调，不引入与现有首页冲突的视觉语言。待办页是管理型界面，因此重点放在：

- 信息层级清晰
- 摘要卡片与筛选区视觉分层明显
- 列表密度高但不拥挤
- 当前导航态、筛选态、状态标签清晰可辨

## 测试策略

第一版至少覆盖以下三层：

### 数据层测试

验证：

- 创建任务
- 更新任务状态
- 根据状态、优先级、分类进行筛选查询

### 服务端动作测试

验证：

- 必填字段校验生效
- 非法状态值不会写入数据库
- 非法优先级不会写入数据库
- 非法分类不会写入数据库

### 页面基础测试

验证 `/todos` 页面在以下场景下能正确渲染：

- 有任务数据
- 无任务数据
- 筛选后无匹配结果

## 实施边界

为了控制首版复杂度，实现阶段只做以下内容：

- 顶部导航接入
- `/todos` 页面 UI
- Prisma 模型与本地 SQLite 持久化
- 创建任务
- 状态切换
- 摘要统计
- 基于 URL query 的筛选

以下能力明确延后：

- 编辑任务标题
- 删除任务
- 批量操作
- 截止日期
- 看板视图
- 用户身份体系

## 设计结论

第一版待办任务页面采用“独立路由 + 顶部双导航 + 控制台列表布局 + Prisma + SQLite”的实现方向。

这套设计满足当前明确需求：

- 保留现有流程图页
- 增加独立待办页
- 支持真实后端存储
- 具备基础项目任务管理能力
- 便于后续在现有模型上继续扩展
